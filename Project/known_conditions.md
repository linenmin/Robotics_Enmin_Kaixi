# Robotics Project 已知条件表

本文档把项目说明、notebook、URDF 和仿真代码里的已知条件统一整理成中文表格。目标是让后续实现时不要反复猜题意。

## 1. 任务一句话

这是一个“反向投篮”任务：球会被抛向机器人，机器人不是投球，而是移动末端的篮筐，让球从篮筐中间穿过去。

小白版理解：

- 球会飞过来。
- 机器人底座不动。
- 机器人有 6 个关节可以转。
- 篮筐固定在机器人末端。
- 我们要写程序，让机器人转动关节，把篮筐中心移到球会经过的位置。

## 5. 球的已知条件

来自 `ball_simulation.py`。


| 条件       | 数值/逻辑                              | 解释                  |
| -------- | ---------------------------------- | ------------------- |
| 球数量      | 默认 `num_balls=1`                   | 默认一次只有一个球           |
| 球半径      | `0.12 m`                           | 题目也给出 120 mm        |
| 初始位置     | `[5.0, 0, 2.0]`                    | 球从远处、高度 2 米的位置出发    |
| 初始竖直速度   | `6.0 m/s`                          | 球一开始向上飞             |
| 重力       | `[0, 0, -9.81]`                    | 球会形成抛物线             |
| 目标区域中心   | `[0.8, 0.15, 0.5]`                 | 发球程序希望球大致飞向机器人附近    |
| 目标区域随机范围 | `goal_area_radius = [0.5, 1.0, 0]` | x/y 有随机偏移，z 固定为 0.5 |
| 位置测量噪声   | 标准差 `1e-3 m`                       | 每次读取球位置有约 1 mm 随机误差 |
| 碰地反弹     | 若 `z < ball_radius`，竖直速度乘 `-0.8`   | 球落地后会反弹并损失能量        |


重要理解：

- 球轨迹不是每次天然固定，因为目标点用了 `np.random.rand()`。
- 如果设置 `np.random.seed(...)`，轨迹可以复现。
- 我们在控制器里拿到的是 `simulation.get_positions()`，也就是带噪声的测量位置，不是真实无噪声位置。

## 7. 仿真循环已知条件

来自 `hoops.ipynb` 的 `simulate()`。


| 条件      | 数值/逻辑                                          | 解释                    |
| ------- | ---------------------------------------------- | --------------------- |
| 控制频率    | `100 Hz`                                       | 每 0.01 秒算一次控制         |
| 控制时间步长  | `dt = 1 / control_freq = 0.01 s`               | 每一步推进 0.01 秒          |
| 渲染频率    | `32 Hz`                                        | 画面大约每秒刷新 32 次         |
| 总仿真时长   | `3 s`                                          | 一次仿真播放 3 秒            |
| 初始关节位置  | `q = robot.q0`                                 | 从固定初始姿态开始             |
| 初始关节速度  | `dq = 0.0`                                     | 代码里写成标量，后续最好改成 6 维零向量 |
| 每步读取    | `ball_positions = simulation.get_positions()`  | 拿到带噪声球位置              |
| 每步控制器输出 | `ddq = controller.step(q, dq, ball_positions)` | 控制器应该输出 6 个关节加速度      |
| 关节积分    | `dq = dq + ddq * dt`，`q = pin.integrate(...)`  | 用加速度更新速度，再更新关节位置      |


注意：当前 notebook 里的 `np.clip(...)` 没有把结果赋值回变量，例如：

```python
np.clip(ddq, -ACCELERATION_LIMIT, ACCELERATION_LIMIT)
```

这行不会真的修改 `ddq`。后续实现时应该写成：

```python
ddq = np.clip(ddq, -ACCELERATION_LIMIT, ACCELERATION_LIMIT)
```

`dq` 和 `q` 的 clip 也同理需要赋值。

## 12. 第一版实现建议

不要一开始就做最复杂的优化。建议分三步。

### Step 1: 只做球轨迹预测

目标：

- 收集最近几帧球位置。
- 用已知重力模型估计球速度。
- 预测未来 1-2 秒球的位置。
- 画出真实测量点和预测曲线。

验收：

- 预测曲线大致跟球飞行方向一致。
- 预测的 z 方向能体现先上升后下降。

### Step 2: 选一个简单接球点

目标：

- 在预测轨迹中找一个高度合适、离机器人不太远的点。
- 先不追求最优，只求能解释、能跑。

验收：

- 接球点在篮筐可达范围附近。
- 接球点不是已经过去的点。

### Step 3: 再做机器人控制

目标：

- 让 `tcp` 向接球点移动。
- 先做简单 task-space 控制或 inverse kinematics。
- 再升级成带约束的优化控制。

验收：

- `tcp` 和目标点距离逐步变小。
- 关节速度/加速度没有超限。
