\section*{Reverse-Hoops-Playing Robot Arm}

\paragraph{Problem setup.}
The project uses a fixed-base UR10 arm that carries a circular hoop at the tool center point. A ball flies through the robot workspace, and the system receives noisy Cartesian ball-position measurements. The simulator specifies independent Gaussian position noise with a standard deviation of 1 mm. The ball has radius \(0.12\,\mathrm{m}\), while the hoop has radius \(0.15\,\mathrm{m}\). This leaves only \(0.03\,\mathrm{m}\) of radial clearance for the ball center when it crosses the hoop plane.

The implementation separates the problem into four steps. The estimator tracks the ball state and predicts a short future trajectory. A simple selector chooses an early candidate on that trajectory. The controller plans a constrained UR10 motion to place the hoop at the candidate. A smarter selector then evaluates multiple candidates with the same controller and chooses the best feasible catch. The robot base remains fixed throughout the project.

\paragraph{Ball trajectory prediction.}
I used a linear Kalman filter because the relevant model is linear under the project assumptions. The state is
\[
x=\begin{bmatrix}p_x&p_y&p_z&v_x&v_y&v_z\end{bmatrix}^\top .
\]
The prediction step integrates constant velocity in \(x,y\) and known gravity in \(z\). The measurement update observes only position, computes the innovation between measured and predicted position, and updates both the state mean and covariance. This choice follows the Bayes-filter structure from the course: prediction propagates the belief through the motion model, while measurement update corrects it using the sensor model. An EKF adds no useful approximation here because the model is already linear. A particle filter also adds unnecessary sampling noise and parameter choices for this low-dimensional Gaussian problem.

Figure~\ref{fig:task1} shows the single-seed trajectory evidence. A 50-seed validation gives mean measurement RMSE \(1.74\,\mathrm{mm}\), mean filtered RMSE \(0.90\,\mathrm{mm}\), and mean 0.5 s future-prediction RMSE \(3.51\,\mathrm{mm}\). The 95th percentile future-prediction RMSE is \(6.35\,\mathrm{mm}\), well below the \(30\,\mathrm{mm}\) hoop tolerance. The covariance matrix remained symmetric and positive semidefinite for all seeds.

\paragraph{Simple interception strategy.}
The simple strategy samples the predicted ball trajectory and applies a small set of geometric filters. It rejects points that are too soon, too low, or outside a rough workspace distance from the current hoop center. It then selects the earliest remaining point. This directly answers the project requirement for a simple interception point on the predicted trajectory. It also keeps the strategy intentionally simple: it does not solve inverse kinematics, it does not estimate control effort, and it does not test the full constrained robot motion.

For the validation seed, the selector chose a future point at \(t=1.36\,\mathrm{s}\) with position \([1.224,-0.119,1.074]\,\mathrm{m}\). It checked 28 predicted points, rejected 6 as too soon, and rejected 21 as too far. The selected point came directly from the predicted trajectory and satisfied the minimum-height filter. This gives a clear baseline for Task 4, where the system replaces the earliest geometric rule with feasibility-aware scoring.

\paragraph{Optimization-based robot control.}
The controller plans a finite-horizon nonlinear program over joint position, velocity, and acceleration:
\[
\{q_0,\ldots,q_N\},\quad \{\dot q_0,\ldots,\dot q_N\},\quad \{\ddot q_0,\ldots,\ddot q_{N-1}\}.
\]
Discrete second-order integration links these variables across the horizon. The optimizer uses hard bounds for joint position, velocity, and acceleration. It uses soft objectives for terminal hoop-center error, velocity regularization, acceleration effort, and hoop orientation. The orientation term penalizes configurations where the hoop top side points toward the ground. This follows the course view of task-space and constraint-based control: the controller treats the catch as the primary task and handles smoothness, limits, and orientation as simultaneous objectives or constraints.

The implementation computes forward kinematics with Pinocchio and solves the horizon problem with CasADi. For the Task 2 target, the NLP reduced the initial hoop-target distance from \(0.731\,\mathrm{m}\) to \(0.020\,\mathrm{m}\). The returned first acceleration command is a finite 6-vector, so it matches the simulator control input. The maximum absolute joint acceleration reached \(1.9996\,\mathrm{rad/s^2}\), and the maximum velocity ratio was \(0.640\). All joint position, velocity, and acceleration bounds passed the validation checks.

Collision handling uses conservative geometric evidence. The planner tracks table clearance for the TCP and selected frames, plus a sphere-based self-clearance proxy. In the Task 3 validation, the minimum TCP-table clearance was \(0.622\,\mathrm{m}\), the minimum frame-table clearance was \(0.115\,\mathrm{m}\), and the minimum self-sphere clearance was \(0.163\,\mathrm{m}\). These values support the planned motion, but they do not prove exact mesh collision freedom.

\paragraph{Smarter interception strategy.}
The smarter selector uses the same predicted trajectory but changes the decision rule. It forms a shared candidate pool, evaluates each shortlisted candidate with the Task 3 NLP, and scores feasible candidates using catch time, terminal error, hoop radial error, acceleration usage, velocity usage, and safety penalties. This makes the Task 4 strategy smarter than Task 2 because it asks whether the robot can actually realize the catch under the controller constraints.

The green ball indicates the physical catch instance. The side-view rendering uses the same viewpoint as the trajectory plot, so the parabolic ball path and the mesh scene describe the same geometry.

The high-score benchmark uses 50 seeds and evaluates all three strategies on the same candidate pool. It defines success by direct hoop-plane crossing: the ball center must cross the hoop plane with radial error at most \(0.03\,\mathrm{m}\). The simple geometric strategy fails in this benchmark because early plausible points often require too much motion. The earliest NLP-feasible baseline reaches 96\% success and catches at \(1.383\,\mathrm{s}\) on average. The smart cost strategy also reaches 96\% success, but it lowers mean radial error from \(10.4\,\mathrm{mm}\) to \(6.13\,\mathrm{mm}\) and reduces mean peak acceleration from \(2.00\) to \(1.50\,\mathrm{rad/s^2}\). It catches later at \(1.429\,\mathrm{s}\), which reveals the central tradeoff: earlier feasible catches cost more control effort, while the smart choice prefers a lower-error and smoother catch.

\paragraph{Results and limitations.}
Table~\ref{tab:summary} summarizes the final evidence. The estimator gives stable short-horizon predictions. The controller respects the joint limits in validation and gives a reproducible trajectory. The Task 4 comparison shows that candidate feasibility matters more than early geometric selection. The best reportable claim is not that the smart selector always catches sooner. The evidence instead shows that it reaches the same success rate as a time-minimizing feasible baseline with lower radial error and lower control usage.

The main limitation is collision modeling. The benchmark records successful smart-cost cases with minimum TCP-table clearance \(0.441\,\mathrm{m}\), minimum frame-table clearance \(0.115\,\mathrm{m}\), and minimum self-sphere clearance \(0.149\,\mathrm{m}\). These checks give useful conservative evidence, but the project implementation does not run full mesh collision checking. The computation also runs offline for planning and evaluation. The simulated catch still respects physical time because the candidate time, ball crossing time, and planned robot trajectory share the same timeline.

\paragraph{Conclusion.}
The final system predicts the ball with a Kalman filter, selects a candidate on the predicted trajectory, plans a constrained UR10 motion, and evaluates smarter candidates through the same controller. The strongest result comes from the feasibility-aware Task 4 benchmark: the smart selector keeps 96\% hoop-crossing success across 50 seeds and reduces both radial error and acceleration demand relative to the earliest feasible baseline. The remaining approximation lies in collision checking, where the report uses explicit clearance metrics rather than claiming full mesh proof.
